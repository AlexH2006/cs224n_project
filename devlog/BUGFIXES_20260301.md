Session focused on fixing runtime failures in `lean_sdpo_goedel_8b_modal.py` when running SDPO training on Modal with PutnamBench problems. Each issue was discovered sequentially by running the training loop end-to-end.

## Issue 1: Verification timeout crashes the entire run

### Symptom
On iteration 2, the Kimina Lean Server took >180s to verify a long proof. The httpx timeout fired, Kimina retried, but the `LeanVerifier` Modal task hit its 300s timeout and raised `FunctionTimeoutError`. This was uncaught, so the whole SDPO run crashed.

### Root cause
Three timeout layers were too tight and misaligned:
- **httpx client timeout** (180s): how long we wait for one `/verify` response from the Lean server.
- **LeanVerifier Modal timeout** (300s): how long the entire `LeanVerifier().verify.remote()` call can run. With one 180s attempt + 3s sleep + start of a second attempt, this was easily exceeded.
- **KiminaLeanServer Modal timeout** (600s): the container hosting the Lean server process.

### Fix
Increased all three timeouts:

| Timeout | Before | After |
|---------|--------|-------|
| httpx client (per-request to Lean) | 180s | 600s |
| LeanVerifier Modal task | 300s | 900s |
| KiminaLeanServer Modal task | 600s | 1200s |

## Issue 2: vLLM rejects LoRA adapters that include embed/lm_head

### Symptom
Iteration 1 worked fine (LoRA v0). On iteration 2, loading LoRA v1 into vLLM failed with:
```
ValueError: vLLM only supports modules_to_save being None.
```
and subsequently:
```
ValueError: base_model.model.lm_head.weight is unsupported LoRA weight
```

### Root cause
`LORA_TARGET_MODULES` included `embed_tokens` and `lm_head`. These are the input embedding table and output projection head â€” not standard linear layers. They can't be decomposed into low-rank A/B matrices. When Unsloth saves the adapter:
1. It puts them in `modules_to_save` in `adapter_config.json` (vLLM rejects this).
2. It saves them as full-weight tensors like `base_model.model.lm_head.weight` in the safetensors file (vLLM can't parse these).

LoRA v0 (saved at setup before any training) happened to work because the weights hadn't been modified yet.

### Fix (clean solution)
Removed `embed_tokens` and `lm_head` from `LORA_TARGET_MODULES` entirely. LoRA now targets only the 7 attention + MLP linear layers (`q_proj`, `k_proj`, `v_proj`, `o_proj`, `gate_proj`, `up_proj`, `down_proj`). The saved adapter is pure LoRA A/B weights that vLLM loads natively â€” no patching needed.

**Earlier approach (superseded):** We initially wrote a `_patch_adapter_config_for_vllm()` function that post-processed each save to null out `modules_to_save` and strip non-LoRA tensors from safetensors files. This worked but was a band-aid â€” the clean fix is to not train those layers with LoRA in the first place.

## Issue 3: Verification hangs on invalid proofs

### Symptom
Iteration 2 generated ~20K chars. The tactic extractor found no valid tactics (just `sorry`), but the code still sent the full proof to Kimina for verification. The Lean server timed out repeatedly, blocking the training loop for 10+ minutes.

### Root cause
The verification call happened **before** the sorry/truncation check:
```python
verification = self._verify_with_retries(full_code)  # blocks for minutes
# ...then checks for sorry after
is_sorry_tactic = tactics.strip().lower() == "sorry"
```

### Fix
Added a quick-reject gate **before** calling Lean verification. Two conditions skip verification entirely:
1. **`is_sorry_tactic`** â€” extracted tactics are just bare `sorry` (nothing to verify).
2. **`is_truncated`** â€” model's `<think>` block never closed (output is garbage).

When either condition is true, a synthetic failed verification is returned immediately with feedback for the teacher prompt. Only proofs with real extracted tactics are sent to the Lean server.

## Issue 4: Verification incorrectly reports `has_sorry` on valid proofs

### Symptom
A proof with no sorry tactics was verified by Kimina as `success: true`, but `LeanVerifier` returned `has_sorry: true` and `complete: false`. This meant the training loop never recognized successful proofs â€” it always computed SDPO loss against them as failures.

### Root cause
`LeanVerifier.verify` used a naive string search as a fallback:
```python
has_sorry = len(sorries) > 0 or "sorry" in lean_code.lower()
```
PutnamBench problems contain `abbrev putnam_1981_a1_solution : â„ť := sorry` in the **problem statement** (not the proof). This matched the string search, so every verification for these problems returned `has_sorry: True` regardless of the actual proof content.

The same pattern appeared in 4 places: the success path, error path, fallback path, and exception path of `LeanVerifier.verify`, plus the quick-reject path in `run_sdpo`.

### Fix
Removed `"sorry" in lean_code.lower()` from all paths. `has_sorry` is now determined **only** by:
- The Kimina server's `sorries` field (reports actual sorry tactics found during type-checking).
- The `is_sorry_tactic` flag in the quick-reject gate (checks if extracted tactics are literally just `"sorry"`).

## Issue 5: CUDA OOM during loss computation on long outputs

### Symptom
Iteration 2 generated 25,428 chars. The forward pass in `_compute_sdpo_loss` OOM'd:
```
torch.OutOfMemoryError: CUDA out of memory. Tried to allocate 198.00 MiB.
GPU 0 has a total capacity of 79.25 GiB of which 4.75 MiB is free.
```

### Root cause
vLLM holds ~32 GB (gpu_memory_utilization=0.4 on 80 GB). After generation, PyTorch's caching allocator holds onto freed memory from vLLM's KV cache and intermediate buffers. The Unsloth forward pass then can't allocate even 198 MiB because the cached (but unused) memory isn't released.

### Fix
Added `torch.cuda.empty_cache()` before the training forward pass in `_compute_sdpo_loss`. This reclaims cached memory from the generation phase (~negligible latency, <10ms). No response token cap is applied â€” the full generated output is used for loss computation.

## Other changes

### Per-iteration debug artifacts
Each iteration now saves its artifacts immediately to the Modal volume under the run directory (`run_{timestamp}/iter_{N}/`), so they survive crashes:
- `raw_output.txt` â€” full raw model output
- `extracted_tactics.txt` â€” what the tactic extractor pulled out
- `full_lean_code.lean` â€” the complete Lean 4 file sent to verification
- `verification.json` â€” the verification result
- `iter_log.json` â€” everything above plus prompts, LoRA version, success flag

The run directory is created at the start of `run_sdpo` (not in `_save_results`), so debug artifacts and final results (logs, metrics, plots, final LoRA) all live under the same `run_{timestamp}/` folder.

### `gradient_accumulation_steps` as CLI parameter
Already existed in `SDPOConfig` and the CLI entrypoint. Confirmed it's fully wired: `--gradient-accumulation-steps 1` updates the model after every problem (vs default 4).
